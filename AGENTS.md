# Repository Instructions

## Player-facing copy

- Write player-facing copy as affirmative, present-tense declarations.
- Center each message on useful game information: the current state, its immediate consequence, or the player's next action.
- Describe active behavior directly. Keep tutorials focused on the mechanic currently visible and the decision currently available.
- Omit meta-commentary about UI behavior, removed behavior, absent events, implementation choices, and contrasts with earlier versions.
- Express empty, idle, unavailable, and destructive states through the state or available action: for example, use “Event log ready,” “Build ready,” or “Deletion clears this slot.”
- Treat words such as “no,” “not,” “never,” “without,” “instead,” “rather,” and future promises about absent behavior as copy-review signals. Rewrite the sentence around the affirmative state whenever domain accuracy permits.
- Keep genuine hazards, losses, costs, and irreversible consequences explicit. State the event and consequence directly, then present the available recovery action.

Before completing a UI copy change, review every modified player-facing string against these rules.
